Engineering Mindset

How I approach
backend systems.

These aren't just principles I've read about — they're patterns I've had to apply and occasionally get wrong in real projects.

🏗

Clean Architecture

Business logic should be isolated from infrastructure. Whether the data source is MySQL or Redis shouldn't change how your domain behaves.

📐

API Design First

Define the contract before writing implementation. Endpoint names, request shapes, and error formats should be intentional, not accidental.

⚡

Performance Awareness

Redis caching, search indexing, lazy loading — not premature optimization, but knowing where the pressure points will be before they become problems.

🔀

Async by Default

If it doesn't need to be synchronous, it shouldn't be. Queuing side effects makes systems more resilient and the API faster without extra complexity.

🗄

Data Modeling Matters

The schema is load-bearing. A well-designed model makes features easier to add and queries more intuitive. I've used relational, document, and graph stores.

🔐

Security by Design

JWT auth, role-based access control, secure token forwarding between services. Security gets designed in — not added as an afterthought.

ClothingStore

Full-stack E-Commerce Platform

Overview

A production-grade e-commerce platform managing product catalog, multi-variant inventory, order processing, and logistics. Led a team of two, responsible for system design and backend implementation.

Key Engineering Decisions

  • → RabbitMQ for async email and inventory sync — keeps the HTTP response fast and the queue resilient
  • → Redis caching for hot product data, reducing repeated DB reads on high-traffic pages
  • → Elasticsearch for full-text search with relevance scoring beyond what SQL LIKE can offer
  • → JWT + Spring Security — stateless auth with role-based access for customers and admins
  • → VNPay + Webhook — idempotent payment confirmation handling to avoid double-processing
  • → GHN API + Webhooks — automated shipping calculation and real-time order tracking

Interesting Problems

Product variant schema

Designing a flexible schema for multi-variant products (size, color, material) without denormalization that breaks querying or pricing logic.

Async inventory consistency

Keeping stock counts consistent when payment confirmation and inventory deduction are decoupled through a message queue.

Elasticsearch–MySQL sync

Reliably keeping the Elasticsearch index in sync with MySQL as the source of truth without expensive full re-indexes.

On the cache layer

Redis sits in front of MySQL for product catalog reads. The cache gets invalidated on inventory changes. For a read-heavy product listing page, this meaningfully reduces database pressure without adding complex cache invalidation logic.

On the queue

Email delivery and inventory sync don't need to happen inside the HTTP request. Pushing them to RabbitMQ means the API responds faster, and if the email service is temporarily down, messages wait in the queue rather than fail silently.

On search

SQL full-text search can work for simple cases, but once you need relevance scoring, fuzzy matching, or faceted filtering by multiple product attributes, Elasticsearch becomes the right tool. The tradeoff is keeping the index synchronized with the MySQL source of truth.

On external APIs

Both VNPay and GHN deliver events via webhooks. Handling payment webhooks required idempotency — the same event could arrive twice, so the system had to recognize and safely discard duplicates.
